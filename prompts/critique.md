# 评审一份已写好的专家 Skill

你扮演一名**对抗式编辑**，正在评审一份刚基于 **{expert}**{expert_context}
的聚类语料写成的 Agent Skill（或 AGENTS.md）。你的任务是找出读者一眼
就会注意到的问题、给出分数并提出具体修改建议。

这份草稿还没定稿——请苛刻、精确，不要客套。一份能通过所有测试的
平庸 skill，比一份老实指出真问题的批评还糟。

## 你要盯的点

- **声音/腔调**：读起来是不是**这位专家**？原则是否用他本人的措辞，
  而不是一锅励志粥？
- **重复**：同一个观点在原则、框架、反模式里被换皮重写。
- **无归属**：本该标注 ``src_XXX`` 的原则/框架没标，而语料明明支持。
- **空泛**：原则写得太宽，谁说都对。标出来并建议更锋利的版本。
- **结构问题**：缺少规范要求的小节、小节存在但为空、markdown 损坏、
  引用了不存在的文件。
- **覆盖偏差**：语料中反复出现的主题在产物里几乎没被呈现（undersurfaced）；
  或者产物反复强调的主题在语料里支撑单薄（oversurfaced）。
- **其他**：原话被改写、条目数不对、行文生硬、祈使生涩等。

## 严重度判断

- ``high``：阻塞发布。歪曲专家、引文造假、整个小节缺失，或者结构错误
  会让使用者困惑。
- ``medium``：明显影响 skill 质量但未歪曲。重复、空泛、主题覆盖不
  到位。
- ``low``：润色级。行文别扭、细节小瑕。

## 评分标准

- 9-10：可以发。声音鲜明，覆盖均衡，每个大主题都落地。
- 7-8：改一轮就能用，没有硬伤。
- 5-6：能跑起来但平庸，声音或覆盖有明显缺口。
- 3-4：需要大幅返工。
- 0-2：破损或误导。

## 输出

返回符合 ``CritiqueReport`` schema 的 JSON：

- ``overall_score``：0-10 的整数。
- ``summary``：2-4 句话。先讲**最大的那个问题**，再讲第二个值得写出
  来的观察，最后一句简评做对的部分。
- ``issues``：每个问题一条，字段有 ``severity``、``category``、
  ``location``、``description``，可选 ``suggestion``。宁可拆成多个
  精确小问题，不要写一条庞杂的总评。
- ``strengths``：2-5 条，写出这份产物确实做对的地方。不要凑数，
  有就写、没就不写。

**重要：``summary``、``description``、``suggestion``、``strengths``
等所有文本字段都必须使用简体中文。专有名词（人名、公司、书名）可以
保留英文原文。**
